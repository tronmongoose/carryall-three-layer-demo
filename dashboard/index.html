Where this fits — post-AIAM

ConductorOne ships two layers today: Baton (the entitlement graph) and AIAM (agent access management, March 2026). Carryall is a prototype that explores a further runtime primitive underneath — per-action capability issuance. Whether that primitive belongs inside AIAM, under it, or nowhere, is an open question I've been sitting with.

Carryall

A per-action capability primitive — cryptographically-bound, TTL-short, auditable per call.
What it explores
  • Intent → minimum-scope envelope

How the prototype consumes Baton

~270-line adapter · 7-method Protocol
Carryall never reaches into Baton's internals. It calls a seven-method Protocol added to authority-runtime in v0.4.0. Any Baton connector plugs in through the same interface with zero Carryall code changes. This is the part of the exercise I'm most confident about — the graph is the right source, and consuming it through a narrow interface works. 
# carryall-baton-backend · BatonBackend implements the Protocol
class BatonBackend:
    def check_access(self, envelope, action, uri) -> Decision:
        # reads the baton graph; returns ALLOW or DENY with reason
        ...

# Carryall side — this is all it knows
backend = load_backend("baton")       # resolved via entry point
decision = backend.check_access(envelope, "read", resource_uri)

Agents

Speculative product surface for a Carryall-style per-action primitive — what it would look like if productized alongside AIAM. Everything below is a design sketch, not a proposal.
What this prototype adds to AIAM
AIAM manages which agents can reach which MCP servers. This prototype explores a finer-grained primitive underneath: a signed, scope-narrowed capability bound to each individual action, not each connection.
Per-action Ed25519 envelopes
Every tool call carries a cryptographically-signed capability with a 300-second TTL, not just a scope check.
Hash-chained audit
Each decision links to the prior one. The evidentiary log is tamper-evident, not just append-only.
Intent → minimum scope
Agent states intent in natural language; PDP narrows to the smallest scope that satisfies it.
Sub-second capability denial
Out-of-scope actions are cryptographically impossible — the capability never compiles — not rejected at the PEP.
Open question: adversarial intent handling is unsolved. AIAM's MCP-server-level granularity sidesteps this problem entirely by not trying to compile intent at all — probably the right call for a platform product.
