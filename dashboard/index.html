Carryall · Policy Decision Point

How Baton and Carryall fit together

Baton is ConductorOne's existing entitlement-graph product. Carryall is the runtime layer on top. The boundary is clean: Baton answers design-time questions; Carryall answers request-time questions.
C1 · today

Baton

The entitlement graph that answers design-time questions about access.
What it powers
  • Extraction from source systems via baton connectors
  • Normalization into v1_resources / v1_entitlements / v1_grants
  • Access reviews, birthright policies, JIT requests
  • Identity lifecycle and audit reporting
Baton answers
Should alice have admin on acme/api?
C1 · proposed

Carryall

The runtime Policy Decision Point that answers request-time questions for every agent action.
What it powers
  • Reading the Baton graph on every agent request
  • Compiling minimum-scope capabilities from stated intent
  • Signing envelopes (Ed25519) with a 300-second TTL
  • Tamper-evident audit log for every issued capability
Carryall answers
Can the release-agent acting as alice audit acme/api right now, for this task?

The handoff

~270-line adapter · 7-method Protocol
Carryall never reaches into Baton's internals. It calls a seven-method Protocol that was added to authority-runtime in v0.4.0. Any Baton connector — GitHub, Okta, Snowflake — plugs in through the same interface with zero Carryall code changes. 
# carryall-baton-backend · BatonBackend implements the Protocol
class BatonBackend:
    def check_access(self, envelope, action, uri) -> Decision:
        # reads the baton graph; returns ALLOW or DENY with reason
        ...

# Carryall side — this is all it knows
backend = load_backend("baton")       # resolved via entry point
decision = backend.check_access(envelope, "read", resource_uri)
